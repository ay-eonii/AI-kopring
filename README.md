## 1. 과제를 어떻게 분석하셨나요?


이번 과제는 긴급 시연이라는 명확한 시간적 제약이 있는 과제라고 판단했습니다.
따라서 완벽보다는 완성을 최우선 목표로 두었습니다.

시연의 핵심은 “API를 통해 AI를 실제로 활용할 수 있음을 보여주는 것”이라고 정의했고,
이를 기준으로 인증 → 대화 생성 → AI 응답 → 데이터 저장까지의 핵심 경로를 먼저 확보했습니다.
반면 분석·보고와 같이 시연 단계에서 트래픽이 높지 않을 것으로 예상되는 메모리 기반으로 우선 구현하였습니다. 

이러한 기준을 바탕으로 제한된 시간 내에 모든 요구사항이 동작 가능한 상태로 완성되도록 우선순위를 조정했습니다.

---

## 2. 과제를 진행함에 있어 AI를 어떻게 활용하셨나요? 어떤 어려움이 있었나요?

AI는 주로 구현 생산성을 높이기 위한 도구로 활용했습니다.
API 요구사항을 기반으로 한 엔드포인트 구조 초안이나, JWT 인증 흐름, 페이지네이션, CSV 응답과 같은 반복적인 보일러플레이트 코드 작성에 있어서는 매우 효율적이었습니다.

다만 단순히 요구사항을 그대로 전달할 경우, AI는 해당 기능이 실제로 사용될 환경이나 비즈니스적인 제약까지는 충분히 고려하지 못한다는 한계가 있었습니다.
예를 들어 AI 호출 시 모든 대화 히스토리를 컨텍스트로 전달하는 방식은 동작 자체는 가능하지만, 토큰 비용 측면에서는 지속적으로 손해가 발생할 수 있는 구조라고 판단했습니다.

그래서 AI가 생성한 결과를 그대로 사용하기보다는, 비용, 응답 시간, 시연 환경을 함께 고려해 최근 대화만 컨텍스트로 사용하는 방식이나
AI 호출 로직을 인터페이스로 분리하는 등 비즈니스적인 판단을 반영해 수정·보완했습니다.

## 3. 지원자가 구현하기 가장 어려웠던 1개 이상의 기능을 설명해주세요.

가장 구현하기 어려웠던 기능은 스트리밍 형태의 응답 전달과 데이터 저장을 동시에 처리하는 로직이었습니다.

시연 환경에서는 실시간 스트리밍 API를 바로 적용하기보다, 하나의 AI 호출로 생성된 응답을 스트리밍처럼 전달하면서 동시에 DB에 저장해야 했습니다.
초기에는 스트리밍 전달과 저장을 분리해 구현하면서, 동일한 질문에 대해 AI API를 두 번 호출하는 비효율적인 구조가 되었습니다.

이를 개선하기 위해, AI 응답을 한 번만 생성한 뒤 이를 청크 단위로 나누어 클라이언트에 전송하고,
동시에 콜백을 통해 응답을 누적해 스트리밍 완료 시점에 한 번만 DB에 저장하도록 변경했습니다.

이 방식으로 API 호출을 한 번으로 줄이면서도 스트리밍 형태의 사용자 경험과 데이터 저장 요구사항을 동시에 만족할 수 있었고,
AI 호출 비용 역시 절반 수준으로 줄일 수 있었습니다.
